# Бриф для дизайнера — приложение для тренера

Версия 1.0 · Платформа: iOS (нативное, SwiftUI) · Ориентация: только портрет

---

## 1. О чём приложение

Личный инструмент **одного тренера**, который ведёт своих подопечных индивидуально.
Сейчас тренер живёт в трёх несвязанных местах: расписание — в Календаре айфона,
программы — в Заметках, а счёт оплаченных занятий держит в голове. Это неудобно,
особенно учёт пакетов («взяла за 10 занятий — сколько осталось?»).

Приложение собирает всё в одном месте: **клиенты, их пакеты/оплаты и расписание**.

### Кто пользователь
- **Только тренер.** Один человек, один телефон.
- Клиенты в приложение НЕ заходят, аккаунтов у них нет.
- Значит: никакой регистрации, входа, ролей. Открыл приложение — сразу работаешь.

### Главные сценарии тренера
1. **Утром:** «Кого я сегодня тренирую и во сколько?»
2. **После занятия:** отметить, что занятие прошло (списать из пакета) — быстро, на ходу.
3. **Регулярно:** «У кого заканчивается пакет? Кому пора напомнить про оплату?»
4. **Новый клиент / пробное:** быстро завести и записать на занятие.
5. **Планирование:** записать клиента на будущие даты → они попадают в Календарь айфона.

---

## 2. Принципы дизайна (читать перед макетами)

1. **Просто как Заметки.** Это требование №1. Не CRM, не «спортивный комбайн».
   Чистый, спокойный, системный вид. Если элемент можно убрать — убираем.
2. **Частые действия — в 1–2 тапа.** Отметить занятие проведённым, продлить пакет,
   завести клиента. Без многошаговых форм и мастеров.
3. **Минимум обязательных полей.** Новый клиент = достаточно имени. Остальное потом.
4. **Главный экран отвечает на 2 вопроса:** «кого тренирую сегодня» и «кому пора платить».
5. **Цифра остатка — всегда на виду.** Остаток занятий по пакету — самый важный элемент
   во всём приложении. Он должен читаться мгновенно везде, где есть клиент.
6. **Большие зоны нажатия.** Тренер пользуется в зале, на ходу, иногда одной рукой.
7. **Не пугать пустотой.** Все пустые состояния — дружелюбные, с понятным первым шагом.

## 3. Навигация

Нижний таб-бар, **2 вкладки** + плавающая кнопка добавления:

```
┌─────────────────────────────┐
│                             │
│        контент экрана       │
│                             │
│                       ( + ) │   ← плавающая кнопка «Добавить»
│                             │
├──────────────┬──────────────┤
│   Сегодня    │   Клиенты    │   ← таб-бар
└──────────────┴──────────────┘
```

- **Сегодня** — расписание/занятия (стартовый экран при запуске).
- **Клиенты** — список клиентов и их пакеты.
- **( + )** — контекстное добавление: на «Сегодня» добавляет занятие,
  на «Клиенты» — клиента. (Дизайнеру: показать оба варианта.)

> Экран «Деньги/Финансы» в V1 НЕ делаем отдельной вкладкой — учёт оплат живёт внутри
> карточки клиента. Заложить место под третью вкладку на будущее, но не рисовать.

## 4. Экраны — детально

### 4.1. Экран «Сегодня»

Назначение: расписание занятий, основное рабочее место тренера в течение дня.

**Шапка**
- Заголовок «Сегодня» + дата (напр. «Сегодня, 27 июня»).
- Переключение дней: свайп влево/вправо или стрелки ‹ ›. Должна быть кнопка
  «вернуться к сегодня», если ушёл на другой день.

**Тело — список занятий дня**, отсортирован по времени. Каждая строка занятия:
- Время (напр. `09:00`).
- Имя клиента (крупно).
- Остаток пакета мелким текстом (напр. `осталось 3 из 10`) + цветовая точка-сигнал.
- Тип занятия, если пробное/разовое — маленький бейдж.
- Статус: запланировано (по умолчанию) / проведено (галочка, приглушённый вид) /
  отменено (зачёркнуто).

**Главное действие на строке** — отметить «Проведено»:
- Свайп по строке вправо → «Проведено» (списывает 1 занятие из пакета).
- Свайп влево → «Отменить» / «Перенести».
- Нужно подтверждение-«отмена действия» (toast «Готово · Отменить»), чтобы случайный
  свайп можно было откатить.

**Особый случай:** если у клиента пакет уже на нуле, а его отмечают «проведено» —
показать мягкое предупреждение «Пакет закончился. Провести в долг / Оформить новый пакет».

**Состояния**
- Обычный день: список занятий.
- Пустой день: иллюстрация + «На этот день занятий нет» + кнопка «Записать занятие».
- Прошедший день: занятия видны, действия доступны (мог не успеть отметить вчера).

---

### 4.2. Экран «Клиенты»

Назначение: видеть всех подопечных и **у кого заканчивается пакет**.

**Шапка**
- Заголовок «Клиенты».
- Поиск по имени.
- Фильтр/сегменты (необязательно, но желательно): Все · Активные · Пробные · Архив.
- Возможна сортировка: «по остатку занятий» (сначала те, у кого мало) — это помогает
  ответить на вопрос «кому пора платить».

**Тело — список клиентов.** Каждая строка (ключевой компонент, см. §5):
- Имя.
- **Остаток занятий — крупно и заметно** (напр. `3 / 10`).
- Цветовой сигнал (🟢/🟡/🔴, см. §6).
- Бейдж статуса для пробных/новых.
- (Опционально) аватар-инициалы.

**Действие:** тап по строке → карточка клиента.

**Состояния**
- Список клиентов.
- Поиск без результатов: «Никого не нашли».
- Совсем пусто (первый запуск): приветствие + «Добавьте первого клиента».

---

### 4.3. Карточка клиента

Назначение: всё об одном человеке — контакты, пакет, история, оплаты.

**Шапка**
- Имя (крупно), статус-бейдж (активный/пробный/новый/архив).
- Кнопки быстрой связи: позвонить / написать в телеграм (если контакт указан).
- Кнопка «Редактировать».

**Блок 1 — Текущий пакет (самый важный)**
- Тип пакета (напр. «Пакет 10 занятий» / «Пробное» / «Разовое»).
- Большой индикатор: **оплачено N · проведено M · осталось K**.
  Можно прогресс-полоской или кольцом. Остаток — самый крупный элемент.
- Дата оплаты и цена.
- Кнопка **«Продлить / новый пакет»** — заметная.
- Если пакета нет / закончился: блок-призыв «Оформить пакет».

**Блок 2 — Заметки и цель**
- Свободный текст (как в Заметках). Цель клиента, травмы, предпочтения.
- Должно быть легко дописывать.

**Блок 3 — Ближайшие занятия**
- Список запланированных занятий этого клиента.
- Кнопка «Записать занятие».

**Блок 4 — История**
- Прошедшие занятия (дата, статус).
- История оплат (когда, какой пакет, сумма) — это и есть учёт денег в V1.

**Действия с клиентом**
- Редактировать, архивировать (не удалять — чтобы история сохранилась), удалить.

---

### 4.4. Форма «Новый / редактировать клиент»

Минимализм. Поля:
- **Имя** — единственное обязательное.
- Телефон / телеграм — необязательно.
- Статус — по умолчанию «активный»; для нового можно «пробный».
- Цель / заметки — необязательно.
- (Опционально на этом же экране) сразу оформить первый пакет — но не заставлять.

Кнопка «Сохранить» активна, как только введено имя.

---

### 4.5. Форма «Оформить / продлить пакет»

Вызывается из карточки клиента. Гибкая, т.к. тренер берёт по-разному:
- **Тип:** Пакет на N занятий · Разовое · Пробное · Абонемент на период.
- **Количество занятий** (для пакета; быстрые пресеты: 5 / 10 / 12 + своё число).
- **Цена.**
- **Дата оплаты** (по умолчанию сегодня).
- Сохранение → пакет становится текущим, остаток = количество.

> Дизайнеру важно: форма должна одинаково спокойно выглядеть и для «10 занятий»,
> и для «пробного на 1». Не строить вокруг числа 10.

---

### 4.6. Форма «Записать занятие»

- Клиент (если зашли из карточки — уже выбран).
- Дата и время.
- (Авто) списывается из текущего пакета клиента.
- Повтор? — на будущее (еженедельно). В V1 можно без повтора, но заложить место.
- Сохранение → создаётся занятие **и событие в Календаре айфона** (см. §7).

---

## 5. Ключевые компоненты (нарисовать отдельно, в разных состояниях)

1. **Строка клиента** (в списке) — имя + остаток `K/N` + цветовой сигнал + бейдж статуса.
2. **Строка занятия** (в расписании) — время + клиент + остаток + статус + свайп-действия.
3. **Карточка пакета** — оплачено/проведено/осталось, прогресс, кнопка продления.
   Варианты: активный пакет, заканчивается, закончился, пробное, разовое.
4. **Бейджи статуса** — активный / пробный / новый / архив.
5. **Цветовой сигнал остатка** — 🟢/🟡/🔴 (см. §6).
6. **Плавающая кнопка «+»** и её контекстные варианты.
7. **Пустые состояния** — для каждого экрана.
8. **Toast подтверждения** с «Отменить».

---

## 6. Логика цветового сигнала (для остатка)

Тренеру нужно мгновенно видеть, кому пора платить:
- 🟢 **Зелёный** — занятий достаточно (по умолчанию > 2).
- 🟡 **Жёлтый** — осталось 1–2 → скоро напомнить про оплату.
- 🔴 **Красный** — 0 / пакет закончился → пора брать деньги.

> Пороги (когда жёлтый/красный) сделаем настраиваемыми в коде — дизайнеру достаточно
> показать три состояния. Уточнить у тренера, удобно ли предупреждать на «2».

---

## 7. Интеграция с Календарём айфона

- Приложение пишет занятия в **Календарь айфона** (отдельный календарь «Тренировки»,
  чтобы не мешать личным событиям). Цвет календаря — можно фирменный.
- Направление одностороннее: приложение → Календарь (создать/изменить/удалить событие).
- Тренер продолжает видеть расписание в привычном Календаре, но ведёт его из приложения,
  где занятие связано с клиентом и оплатой.
- При первом запуске — экран-объяснение и системный запрос доступа к Календарю.
  Дизайнеру: нарисовать дружелюбный «pre-permission» экран («Зачем доступ к Календарю»)
  перед системным диалогом.

## 8. Визуальное направление

- **Тон:** спокойный, чистый, «эппловский». Ближе к Заметкам / Напоминаниям, чем к
  ярким фитнес-приложениям. Никаких агрессивных градиентов и тёмного «качалочного» стиля.
- **Шрифты:** системный San Francisco (родной iOS). Крупные читаемые заголовки.
- **Цвет:** в основном нейтральная светлая база + один акцентный цвет бренда +
  семафор 🟢🟡🔴 строго для сигнала остатка (больше нигде, чтобы не путать).
- **Тёмная тема:** заложить (iOS-пользователи ждут). Можно отдельной итерацией.
- **Иконки:** SF Symbols (родная библиотека Apple) — единый стиль, экономит время.
- **Скругления/отступы:** системные iOS-стандарты (крупные карточки, воздух).

## 9. Крайние случаи и состояния (не забыть)

- Клиент без пакета вообще.
- Пакет на нуле, но клиента отмечают «проведено» (в долг).
- Клиент с несколькими пакетами в истории (показываем текущий + историю).
- Очень длинное имя / длинная заметка.
- Пустой сегодняшний день / пустой список клиентов / пустой поиск.
- Нет доступа к Календарю (тренер отказал) — приложение работает, но без записи в Календарь;
  показать ненавязчивую подсказку, как включить.
- Архивный клиент (скрыт из основного списка, история сохранена).

---

## 10. Что НЕ входит в V1 (не рисовать)

- Аккаунты/вход для клиентов, клиентское приложение.
- Сервер, облако, чаты.
- Программы тренировок и база упражнений (остаются в Заметках; это V2).
- Приём онлайн-оплат внутри приложения (учёт денег — да, эквайринг — нет).
- Полноценная бухгалтерия (расходы, налоги).
- Двусторонняя синхронизация с Календарём.
- Push-уведомления (V2).

---

## 11. Что нужно от дизайнера на выходе

1. Макеты всех экранов из §4 в основных состояниях (обычное + пустое + крайние).
2. Все ключевые компоненты из §5 в их вариантах состояний.
3. Светлая тема обязательно; тёмная — желательно (можно второй итерацией).
4. Экраны флоу целиком (кликабельный прототип приветствуется):
   - Завести нового клиента → оформить пакет → записать на занятие.
   - Утренний просмотр «Сегодня» → отметить занятие проведённым.
   - Увидеть 🔴 у клиента → открыть карточку → продлить пакет.
5. Иконка приложения.
6. Палитра, типографика, отступы — в виде простого набора стилей/токенов.

## 12. Открытые вопросы к тренеру (уточнить до финальных макетов)

1. На каком остатке удобно предупреждать об оплате — на 2 занятиях или раньше?
2. Нужен ли отдельный экран «доход за месяц» в V1, или достаточно истории оплат в карточке?
3. Как обычно записывает контакт клиента — телефон, телеграм, и то и другое?
4. Бывают ли групповые занятия (несколько клиентов в одно время), или только индивидуально?
   (От этого зависит модель занятия — уточнить заранее.)
